## 1. Final MVP Scope

### What MVP Delivers

- Bilingual (FR primary / EN secondary) website, statically generated with Next.js App Router
- France-based brand, EUR pricing, international shipping structure prepared in content
- 12 pages fully built (including Mentions légales, required by French law)
- Homepage with 10 conversion sections
- Routine pack page as the **primary commercial destination**
- Product detail pages and shop page
- Order form per product/pack → Next.js API route → Resend email (business notification + customer confirmation)
- Inline success state after order submission (no redirect, no separate success page)
- WhatsApp CTA as a co-primary conversion path — not the only path
- SEO metadata and JSON-LD structured data on all pages
- Legal/compliance pages prepared for French law and GDPR — all content flagged for legal review before launch
- Anti-spam on order form: Zod server validation + honeypot field + optional Upstash rate limiting
- Mobile-first responsive design deployed to Vercel

### What MVP Does NOT Deliver

See [Section 2](#2-exclusions-from-mvp) for the full exclusions list.

  seoDescription: LocalizedString

## 7. Order Flow Architecture

### Flow Diagram

```
User fills OrderForm (product pre-selected from page context)
         ↓
Client-side Zod validation (inline field errors)
         ↓
POST /api/order
{
  name, email, phone, country, address,
  product, quantity, message?,
  consent: true,
  _hp: ""   ← honeypot — must be empty
}
         ↓
Server: Zod schema validation
         ↓  fail → 400 { error: "..." }
Server: Honeypot check (_hp must be "" — bots fill it)
         ↓  fail → 400 silently (do not confirm spam detection)
Server: Rate limit check (if Upstash env vars are present)
         ↓  fail → 429 { error: "Too many requests" }
         ↓  pass (or Upstash not configured — skip gracefully)
Resend: send order notification to BUSINESS_EMAIL
Resend: send confirmation email to customer
         ↓
Return 200 { success: true }
         ↓
OrderForm renders inline success state:
  - Confirmation message
  - Summary of what was ordered
  - Next steps ("We will contact you to confirm your order and arrange delivery")
  - WhatsApp follow-up link
```

### Rate Limiting Behaviour

Rate limiting is **optional** in MVP. The `/api/order` route checks for the presence of `UPSTASH_REDIS_REST_URL` and `UPSTASH_REDIS_REST_TOKEN` environment variables before attempting to use Upstash.

```
If Upstash env vars present:
  → apply rate limiting (e.g. 5 submissions per IP per 10 minutes)
If Upstash env vars missing:
  → skip rate limiting silently
  → Zod validation and honeypot remain active
  → API route does not crash
```

`lib/rate-limit.ts` encapsulates this conditional logic. The `/api/order` route calls `checkRateLimit(ip)` and receives either `{ limited: false }` or `{ limited: true }` — it never throws.

### Anti-Spam Summary

| Layer | Always active | Notes |
|---|---|---|
| Zod server validation | Yes | Rejects malformed payloads |
| Honeypot field | Yes | `_hp` must be empty — hidden from real users |
| Upstash rate limiting | Optional | Activated by env vars, skipped if absent |
| Vercel DDoS protection | Yes | Platform-level, always on |

### Environment Variables

```
RESEND_API_KEY=                     # required
BUSINESS_EMAIL=orders@pureva.fr     # required
FROM_EMAIL=noreply@pureva.fr        # required
NEXT_PUBLIC_WHATSAPP_NUMBER=33XXXXXXXXX   # required
NEXT_PUBLIC_SITE_URL=https://pureva.fr    # required
UPSTASH_REDIS_REST_URL=             # optional — enables rate limiting
UPSTASH_REDIS_REST_TOKEN=           # optional — enables rate limiting
```

### Email Contents

**Order notification to business:**
Product name, quantity, customer name, email, phone, country, delivery address, optional message, submission timestamp.

**Confirmation to customer:**
Order received confirmation, product ordered, next steps, WhatsApp link for follow-up questions.

## 10. FR/EN i18n Approach

### Library

`next-intl` v3 — App Router native, type-safe message access, minimal bundle impact.

### Structure

- Default locale: `fr` — root `/` redirects to `/fr`
- Supported locales: `['fr', 'en']`
- `middleware.ts` handles browser locale detection (Accept-Language) and enforces `[locale]` prefix
- All UI strings in `src/i18n/messages/{locale}.json`
- Product-specific copy (names, descriptions, benefits, how-to-use, WhatsApp messages) stored **inline in product data** as `Record<Locale, string>` — avoids scattering product content across JSON files

### Language Switcher

Renders in `Header`. Uses `next-intl`'s `useRouter` and `usePathname` to navigate to the same path under the alternate locale. Example: `/fr/shop` → `/en/shop`.

### hreflang

Injected via `alternates.languages` in `generateMetadata()` on every page. Tells search engines which page is the FR vs EN canonical version.

### Localized Slugs

Each product has `slug: { fr: '...', en: '...' }`. The `[slug]/page.tsx` uses `generateStaticParams()` to emit both FR and EN paths at build time. Both are statically generated.

## 11. France-Based Legal / Compliance Structure

> **IMPORTANT:** These pages prepare the structural and content framework for legal compliance. The actual legal text must be reviewed and validated by a qualified professional before public launch. All legal page templates ship with explicit `{/* TODO: legal review required before launch */}` markers. The site does not automatically become legally compliant by having these pages.

### Required Pages

| Page | Route | Key required content |
|---|---|---|
| **Mentions légales** | `/fr/legal-notice` | Publisher identity (name, address, SIRET if applicable), host details (Vercel Inc.), responsible editor name |
| **CGV / Terms** | `/fr/terms` | Sale conditions, prices in EUR incl. applicable VAT, 14-day withdrawal right (droit de rétractation), delivery terms, dispute resolution, applicable law: French law |
| **Privacy / RGPD** | `/fr/privacy` | Data collected (name, email, phone, address via order form), purpose (order fulfilment), legal basis (contract performance), data processor (Resend for email delivery), retention period, user rights (access, rectification, deletion, portability), contact email for requests, no data sold to third parties |
| **Shipping & Returns** | `/fr/shipping-returns` | Zone-based delivery estimates, return window, return process, contact method |

English equivalents of all legal pages are provided under `/en/` routes. Content structure mirrors the FR version; specific legal obligations under French law are annotated as FR-specific where relevant.

### Mentions Légales Specifics

Required by French law (Loi pour la Confiance dans l'Économie Numérique — LCEN). Must include:

## 12. International Shipping Readiness

Shipping zones are content definitions only. No shipping logic is computed in MVP. These definitions are used on the Shipping & Returns page and are structured to feed a future checkout shipping calculator.

### Shipping Zone Type

```typescript
// src/types/shipping.ts

export type ShippingZone = {
  id: string
  name: LocalizedString
  countries: string[]                  // ISO 3166-1 alpha-2 codes
  estimatedDelivery: LocalizedString   // '2-3 jours ouvrés' / '2-3 business days'
  note?: LocalizedString               // e.g. free shipping threshold note
  freeShippingThreshold?: number       // EUR cents; optional
}
```

### Initial Zones (content only — not enforced by logic in MVP)

| Zone | Countries | Delivery estimate |
|---|---|---|
| France métropolitaine | FR | 2–3 business days |
| Union Européenne | EU27 | 5–10 business days |
| International | All others | 10–20 business days |

Notes for display on Shipping & Returns page:
- International customers are responsible for any applicable customs duties
- Free shipping threshold (if offered) configured per zone via `freeShippingThreshold`
- EU VAT OSS compliance is a legal/financial matter for the business owner, not handled in MVP

### Future Readiness

The `ShippingZone` type and data structure are designed so a Phase 1.5 Stripe checkout can read these zones to compute shipping cost and delivery estimate per country. The product data model's `price` in EUR cents is directly compatible with Stripe's amount format.

## 14. Stripe / Payment Roadmap

MVP ships with no payment processing, no Stripe packages, no Stripe routes, and no placeholder files for Stripe. Stripe is a Phase 1.5 addition.

### What MVP Prepares (without implementing)

- `price` and `compareAtPrice` stored in EUR cents — exactly the format Stripe expects
- `lib/format-price.ts` handles display formatting — no changes needed when Stripe is added
- `OrderForm` accepts a `mode` prop: `'inquiry'` (MVP) | `'checkout'` (Phase 1.5)
- `ShippingZone` data structure is ready to feed a shipping cost calculator
- No pages or API routes are created as placeholders

### Phase 1.5 Stripe Integration Plan (contained — no existing files rewritten)

1. Add `stripe` and `@stripe/stripe-js` packages
2. Create `/api/checkout` POST route: receives product + quantity → creates Stripe Checkout Session → returns `session.url`
3. `OrderForm` in `'checkout'` mode POSTs to `/api/checkout` and redirects to Stripe-hosted page
4. Add `/[locale]/order-success` and `/[locale]/order-cancelled` pages
5. Add `/api/webhooks/stripe` handler for `checkout.session.completed` → send order confirmation emails via Resend
6. Add `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, `NEXT_PUBLIC_STRIPE_PUBLISHABLE_KEY` to env vars

No cart is required: Stripe Checkout handles multi-product orders via line items.

## 16. Risks and Assumptions

| Risk / Assumption | Status | Mitigation |
|---|---|---|
| No product photos at build start | Known | Placeholder images used; design is photo-ready — image swap requires no code change |
| No real reviews at launch | Known | `ReviewsSection` returns `null` if array empty — never an empty shell, never fake content |
| Legal page content | Risk | Templates ship with `TODO: legal review required` markers. Must be validated by a qualified professional familiar with French consumer law (LCEN, CGV, RGPD) before launch |
| Mentions légales requires real SIRET / identity | Assumption | Owner must provide publisher identity before launch — page is structurally complete |
| Localized slugs are permanent | Risk | Canonical slugs defined in this spec are final post-launch. URL changes after indexing require 301 redirects |
| Resend free tier volume | Low risk | 3,000 emails/month — sufficient for MVP; upgrade is trivial and non-breaking |
| Rate limiting absent (Upstash not configured) | Accepted | Honeypot + Zod + Vercel DDoS protection are sufficient at MVP order volume |
| Stripe not in MVP | Confirmed | Price-in-cents model and `OrderForm` mode prop make Phase 1.5 a contained addition |
| EU VAT on cross-border B2C sales | Risk (financial/legal) | Not handled in MVP. EU OSS registration may be required once sales thresholds are met — flagged for owner |
| WhatsApp number change post-launch | Low risk | Stored as `NEXT_PUBLIC_WHATSAPP_NUMBER` env var — change requires Vercel env update and redeploy only |
| Disclaimer visibility | Non-negotiable | `Disclaimer` is a mandatory component; it must appear on every product page and routine pack page without exception — never conditional, never hidden |
| i18n adds URL complexity | Managed | `next-intl` handles cleanly; the only ongoing overhead is keeping FR and EN message files in sync |
